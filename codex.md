## 时间复杂度

先看一个最熟悉的例子：**线性查找**。

```java
/**
 * 线性查找
 *
 * @param a      待查找的数组（可以无序）
 * @param target 待查找的目标值
 * @return 找到则返回索引，找不到返回 -1
 */
public static int linearSearch(int[] a, int target) {
    for (int i = 0; i < a.length; i++) {
        if (a[i] == target) {
            return i;
        }
    }
    return -1;
}
```

### 先问一个问题

这段代码和二分查找一样，**都能找到目标值**。
那它们的差别到底在哪里？

答案是：

> **差别不在“能不能找到”，而在“数据量变大后，要花多少时间才能找到”。**

这正是时间复杂度要衡量的事。

---

### 1）线性查找的执行次数

先看线性查找：

```java
/**
 * 线性查找
 *
 * @param a      待查找的数组（可以无序）
 * @param target 待查找的目标值
 * @return 找到则返回索引，找不到返回 -1
 */
public static int linearSearch(int[] a, int target) {
    for (int i = 0; i < a.length; i++) {
        if (a[i] == target) {
            return i;
        }
    }
    return -1;
}
```

考虑最坏情况：**数组中没有目标值**。
例如在 `[1, 2, 3, 4]` 中查找 `5`。

此时程序会从头走到尾，把每个元素都检查一次。

如果数组长度是 `n`，可以把各部分执行次数拆开来看：

* `int i = 0`
  只会执行 **1 次**

* `i < a.length`
  每次进入循环前都要判断一次，循环结束时还会再判断一次，所以总共执行 **`n + 1` 次**

* `i++`
  每轮循环结束执行一次，共执行 **`n` 次**

* `a[i] == target`
  每轮循环都会比较一次，共执行 **`n` 次**

* `return i`
  因为这里讨论的是**最坏情况：没找到**，所以这一行执行 **0 次**

* `return -1`
  循环结束后执行 **1 次**

把它们加起来：

$$
1 + (n+1) + n + n + 1
$$

整理后得到：

$$
3n + 3
$$

也就是说，如果把每一步执行时间粗略看成一样，那么线性查找的总执行次数可以写成：

$$
T(n) = 3n + 3
$$

---

#### 从 `3n + 3` 到 `O(n)`

这里最重要的不是常数 `3` 和 `+3`，而是看它随着 `n` 增大时的**增长趋势**。

因为：

* `3n` 会随着 `n` 变大而变大
* `+3` 只是固定常数，影响很小
* 常数倍 `3` 不改变增长等级

所以当 `n` 很大时，`3n + 3` 的主要趋势和 `n` 是一致的。

因此：

$$
T(n) \in O(n)
$$

也就是说，**线性查找的时间复杂度是 `O(n)`**。

---

#### 这一段真正想说明什么？

这段推导不是要你死背 `3n + 3`，而是要你看懂两件事：

1. 可以从代码中拆出“每一部分执行了几次”
2. 最后复杂度关注的是**增长趋势**，不是精确常数

所以线性查找的核心特征是：

> 数据规模变成 2 倍，最坏情况下要检查的元素数量也大致变成 2 倍。

这就是为什么它属于 **线性时间复杂度 `O(n)`**。

---

### 2）二分查找的执行次数

再看二分查找：

```java
/**
 * 二分查找基础版
 *
 * @param a      待查找的升序数组
 * @param target 待查找的目标值
 * @return 找到则返回索引，找不到返回 -1
 */
public static int binarySearchBasic(int[] a, int target) {
    int i = 0, j = a.length - 1;
    while (i <= j) {
        int m = (i + j) / 2;
        if (target < a[m]) {
            j = m - 1;
        } else if (a[m] < target) {
            i = m + 1;
        } else {
            return m;
        }
    }
    return -1;
}
```

二分查找和线性查找最大的不同，不在于“能不能找到”，而在于：

> **它每比较一次，就能把查找范围缩小一半。**

---

#### 先看循环会执行多少次

假设数组长度是 `n`。

第一次查找时，搜索范围有 `n` 个元素。
比较一次后，只会留下其中一半：

$$
n \rightarrow \frac{n}{2}
$$

再比较一次后，范围又缩小一半：

$$
\frac{n}{2} \rightarrow \frac{n}{4}
$$

继续下去会变成：

$$
n,\ \frac{n}{2},\ \frac{n}{4},\ \frac{n}{8},\ \dots
$$

直到范围缩小到只剩 `1` 个元素为止。

假设总共循环了 `L` 次，那么大致会满足：

$$
\frac{n}{2^L} = 1
$$

移项后得到：

$$
2^L = n
$$

所以：

$$
L = \log_2 n
$$

##### 什么是 `log_2 n`？

这里的 `log_2 n` 表示：

> **把数据范围不断除以 2，要除多少次，才会缩小到 1。**

因为二分查找每执行一轮，都会把查找范围缩小一半，所以循环次数就会和 $\log_2 n$ 有关。

例如：

* 当 `n = 8` 时：

  $$
  8 \rightarrow 4 \rightarrow 2 \rightarrow 1
  $$

  一共缩小了 `3` 次，所以：

  $$
  \log_2 8 = 3
  $$

* 当 `n = 16` 时：

  $$
  16 \rightarrow 8 \rightarrow 4 \rightarrow 2 \rightarrow 1
  $$

  一共缩小了 `4` 次，所以：

  $$
  \log_2 16 = 4
  $$

也可以反过来理解：

* $2^3 = 8$，所以 $\log_2 8 = 3$
* $2^4 = 16$，所以 $\log_2 16 = 4$

这表示：

> 二分查找的循环次数不是 `n` 级别，而是 $\log_2 n$ 级别。

在实际代码里，因为还要考虑“最后一次判断失败才退出循环”，以及 `n` 不一定刚好是 `2` 的整数次幂，所以更严谨一点可以写成：

$$
L \approx \lfloor \log_2 n \rfloor + 1
$$

但在复杂度分析时，我们只需要抓住核心：

> **循环次数的增长趋势是 $\log n$。**

---

#### 把各部分执行次数拆开来看

为了和前面的线性查找保持同样的分析方式，我们也把代码逐项拆开。

设二分查找的循环次数为 `L`。

##### 1. 初始化部分

```java
int i = 0, j = a.length - 1;
```

这里是初始化左右边界，只执行 **1 次**。

---

##### 2. 循环条件判断

```java
while (i <= j)
```

每进入一轮循环前都要判断一次，循环结束时还会再多判断一次，所以总共执行：

$$
L + 1
$$

次。

---

##### 3. 计算中间位置

```java
int m = (i + j) / 2;
```

每轮循环都会计算一次中间索引，所以执行：

$$
L
$$

次。

---

##### 4. 第一层比较

```java
if (target < a[m])
```

每轮循环都一定会先判断这一句，所以执行：

$$
L
$$

次。

---

##### 5. 第二层比较

```java
else if (a[m] < target)
```

这一句不一定每轮都会执行。

因为如果第一层条件 `target < a[m]` 已经成立，就不会再进入 `else if`。
但如果我们讨论的是最坏情况，可以粗略认为这一句在大多数轮次里也会被判断，所以可以把它也看成执行约：

$$
L
$$

次。

> 这里不是在追求精确常数，而是在看整体增长趋势。
> 就算这一项不是刚好 `L` 次，而是少几个常数，也不会影响最后的复杂度等级。

---

##### 6. 更新边界

```java
j = m - 1;
```

或

```java
i = m + 1;
```

每轮循环至多只会执行其中一个，因此这一类“边界更新操作”总共执行大约：

$$
L
$$

次。

---

##### 7. 返回结果

如果讨论的是**最坏情况：没找到**，那么：

```java
return m;
```

执行 `0` 次。

而：

```java
return -1;
```

在循环结束后执行 **1 次**。

---

#### 把它们加总起来

把上面的主要操作加起来，可以得到一个近似式：

$$
1 + (L + 1) + L + L + L + L + 1
$$

整理后大约是：

$$
5L + 3
$$

再把 `L` 代入：

$$
L \approx \log_2 n
$$

于是总执行次数可以写成：

$$
T(n) \approx 5\log_2 n + 3
$$

如果写得更严谨一点，也可以写成：

$$
T(n) \approx 5(\lfloor \log_2 n \rfloor + 1) + 3
$$

---

#### 从 `5log₂n + 3` 到 `O(log n)`

这里最重要的，仍然不是常数 `5` 和 `+3`，而是看它的**增长趋势**。

因为：

* $\log_2 n$ 会随着 `n` 变大而变大
* 常数倍 `5` 不影响复杂度等级
* 常数项 `+3` 可以忽略
* 对数的底数在复杂度分析里通常也忽略不写

所以最终可以写成：

$$
T(n) \in O(\log n)
$$

也就是说，**二分查找的时间复杂度是 $O(\log n)$**。

---

#### 这一段真正想说明什么？

这段推导最核心的不是让你背 $5\log_2 n + 3$，而是要理解：

1. 二分查找每次都会把范围缩小一半
2. 所以循环次数不会随着 `n` 线性增加
3. 它的增长速度是 $\log n$ 等级，而不是 `n` 等级

这也是为什么当数据量很大时，二分查找通常会比线性查找快很多。

例如：

* 线性查找在最坏情况下，要一个一个看，可能要看 `n` 次
* 二分查找每次砍半，只需要大约 $\log_2 n$ 次

所以它们虽然都能找到答案，但增长趋势完全不同。

---

### 3）为什么二分查找快很多？

当 `n` 很小时，线性查找和二分查找看起来差距不一定很明显。

但当 `n` 很大时，差距会迅速拉开。

例如：

* 如果 `n = 1024`
* 线性查找最坏情况大约要比较 `1024` 次
* 二分查找最坏情况大约只要比较 `10` 次，因为：

$$\log_2(1024) = 10$$

这就是为什么：

> **线性查找和二分查找都能完成任务，但随着数据量变大，二分查找会明显更有效率。**

---

### 4）什么是时间复杂度？

在计算机科学中，**时间复杂度**是用来衡量：

> 一个算法的执行时间，如何随着数据规模 `n` 的增大而增长。

注意，时间复杂度关心的不是：

* 这台电脑跑了几毫秒
* 你的 CPU 比别人快多少
* 某一行代码到底是 3 次还是 5 次

它更关心的是：

> **增长趋势**

也就是：

* 是不是随着 `n` 线性增长？
* 是不是随着 $\log n$ 增长？
* 是不是随着 `n^2` 增长？

---

## 大 O 表示法

假设一个算法的执行次数可以写成函数：

$$f(n)$$

例如：

* 线性查找可以写成：$f(n) = 3n + 3$
* 二分查找可以写成：$f(n) = 5(\lfloor \log_2 n \rfloor + 1) + 4$

但在分析复杂度时，我们通常不会保留这么细的常数和低阶项，而是只保留**最主要的增长趋势**。

这时就会用到大 O 表示法。

### 化简原则

已知 `f(n)`，想找一个和它增长趋势一致的 `g(n)` 时，通常遵循下面几条规则：

#### 1. 忽略常数倍

例如：

$$100n^2 \Rightarrow n^2$$

因为常数 `100` 不会改变增长等级。

#### 2. 忽略低阶项

例如：

$$n^2 + n \Rightarrow n^2$$

$$n^3 + n^2 + n \Rightarrow n^3$$

因为当 `n` 很大时，高阶项主导整体增长。

#### 3. 对数的底数通常省略

例如：

$$\log_2 n,\ \log_{10} n$$

在复杂度分析里通常都写成：

$$\log n$$

因为不同底数之间只差一个常数倍，而常数倍会被忽略。

---

### 什么是对数？

如果你对 $\log n$ 不熟，先记住下面这句话就够了：

> **对数表示：一个数要连续“除以某个底数”多少次，才会缩小到 1。**

例如：

$$\log_2 8 = 3$$

因为：

$$8 \div 2 \div 2 \div 2 = 1$$

也可以反过来理解为：

$$2^3 = 8$$

所以：

$$\log_2 8 = 3$$

再例如：

$$\log_2 1024 = 10$$

因为：

$$2^{10} = 1024$$

这也是为什么二分查找面对 `1024` 个元素时，大约只要查 `10` 次。

---

## 渐进上界、下界与紧界

这一节最容易混淆，所以先讲一句最重要的：

> `O`、`Ω`、`Θ` 本质上描述的是**增长速度的界**，不是直接等于“最差情况 / 最佳情况”这四个字。

是否是在描述最差情况或最佳情况，要看你前面定义的 `f(n)` 到底代表什么。

例如：

* 如果 `f(n)` 表示**最坏情况执行时间**

  * 那我们常说最坏情况是 `O(...)`
* 如果 `f(n)` 表示**最好情况执行时间**

  * 那也可以分析它的 `Ω(...)` 或 `Θ(...)`

所以请把这件事分开记：

* `O`：上界
* `Ω`：下界
* `Θ`：同阶紧界

---

### 渐进上界 `O(g(n))`

渐进上界（asymptotic upper bound）的定义是：

> 存在正常数 `c` 和 `n_0`，使得当 `n >= n_0` 时，总有：

$$f(n) \le c \cdot g(n)$$

记作：

$$f(n) \in O(g(n))$$

意思是：

> 从某个规模开始，`f(n)` 的增长速度不会超过 `g(n)` 的常数倍。

#### 直觉理解

像你估计通勤时间时说：

* 「最多 60 分钟会到」

这是一个**上界**。

它不表示你每次都要 60 分钟，
而是表示：**不会明显超过这个等级**。

---

### 渐进下界 `Ω(g(n))`

渐进下界（asymptotic lower bound）的定义是：

> 存在正常数 `c` 和 `n_0`，使得当 `n >= n_0` 时，总有：

$$f(n) \ge c \cdot g(n)$$

记作：

$$f(n) \in \Omega(g(n))$$

意思是：

> 从某个规模开始，`f(n)` 的增长速度至少和 `g(n)` 同阶，低不到哪里去。

#### 直觉理解

像你说：

* 「再快也要 15 分钟」

这是一个**下界**。

---

### 渐进紧界 `Θ(g(n))`

渐进紧界（asymptotic tight bound）的定义是：

> 存在正常数 `c_1`、`c_2` 和 `n_0`，使得当 `n >= n_0` 时，总有：

$$c_1 g(n) \le f(n) \le c_2 g(n)$$

记作：

$$f(n) \in \Theta(g(n))$$

意思是：

> 从某个规模开始，`f(n)` 的增长速度和 `g(n)` 是同一个等级。

#### 直觉理解

像你说：

* 「通勤时间大概就是 30 分钟这个等级」

意思不是每次都刚好 30 分钟，
而是说它不会远小于这个量级，也不会远大于这个量级。

---

## 用同一个例子看懂 `O / Ω / Θ`

假设某段程序的执行次数为：

$$f(n)=3n^2 + 5n + 10$$

我们猜它的增长趋势像：

$$g(n)=n^2$$

---

### 1）证明 $f(n) ∈ O(n^2)$

我们要找：

$$f(n) \le c \cdot n^2$$

当 `n >= 1` 时：

* $5n <= 5n^2$
* $10 <= 10n^2$

所以：

$$3n^2 + 5n + 10 \le 3n^2 + 5n^2 + 10n^2 = 18n^2$$

因此取：

* $c = 18$ 
* $n_0 = 1$

就有：

$$f(n) \in O(n^2)$$

---

### 2）证明 $f(n) ∈ Ω(n^2)$

我们要找：

$$f(n) \ge c \cdot n^2$$

因为：

$$3n^2 + 5n + 10 \ge 3n^2$$

恒成立。

所以取：

* $c = 3$
* $n_0 = 1$

就有：

$$f(n) \in \Omega(n^2)$$

---

### 3）证明 $f(n) ∈ Θ(n^2)$

因为它同时满足：

$$f(n) \in O(n^2)$$

以及

$$f(n) \in \Omega(n^2)$$

所以：

$$f(n) \in \Theta(n^2)$$

也就是说：

> $3n^2 + 5n + 10$ 的增长等级就是 $n^2$。

---

## 常见大 O 表示法

常见时间复杂度从低到高大致如下：

* $O(1)$：常数时间
* $O(log n)$：对数时间
* $O(n)$：线性时间
* $O(n log n)$：拟线性时间
* $O(n^2)$：平方时间
* $O(2^n)$：指数时间
* $O(n!)$：阶乘时间

### 如何直觉理解？

* $O(1)$：数据变多，耗时几乎不变
* $O(log n)$：数据变很多，耗时增加得很慢
* $O(n)$：数据翻倍，耗时大致也翻倍
* $O(n^2)$：数据翻倍，耗时可能接近四倍
* $O(2^n)$ / $O(n!)$：增长极快，通常很难处理大规模数据

---

## 空间复杂度

除了时间成本，算法还会消耗空间。

**空间复杂度**衡量的是：

> 一个算法在执行过程中，随着数据规模 `n` 增长，所需要的**额外空间**如何变化。

注意这里的关键词是：

> **额外空间**

也就是说：

* 输入数组本身通常不算进额外空间
* 我们主要看算法为了执行而另外申请了多少空间

例如：

```java
public static int binarySearchBasic(int[] a, int target) {
    int i = 0, j = a.length - 1;
    while (i <= j) {
        int m = (i + j) >>> 1;
        if (target < a[m]) {
            j = m - 1;
        } else if (a[m] < target) {
            i = m + 1;
        } else {
            return m;
        }
    }
    return -1;
}
```

这段代码额外使用了几个变量：

* `i`
* `j`
* `m`

虽然你也可以把它们理解成占用了固定几个字节，
但在空间复杂度里，更重要的不是精确的 byte 数，而是：

> **这些变量的数量不会随着 `n` 增长而增加。**

所以它的额外空间是常数级别，空间复杂度为：

$$O(1)$$

---

## 二分查找性能

下面总结二分查找的性能。

### 时间复杂度

* **最坏情况**：`O(log n)`
* **最好情况**：`O(1)`
  如果第一次就刚好命中中间元素，只需要比较一次

### 空间复杂度

* 只使用了常数个额外变量 `i`、`j`、`m`
* 不会随着输入规模增长而增加

所以空间复杂度是：

$$O(1)$$
