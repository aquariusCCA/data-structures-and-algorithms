---

## 本节导读

这一节讲的是 **二分查找的平衡版**。

它和基础版、改动版最大的不同，不在于“还能不能二分”，而在于：

> **它不急着在循环内直接返回答案，而是尽量让每一轮判断更均衡，先把区间缩到只剩一个候选元素，最后再统一判断。**

所以平衡版的关键不是“写法变难了”，而是思路变了：

* 基础版：每一轮都试图直接找到答案
* 平衡版：循环里主要负责 **缩小区间**
* 等区间缩到只剩一个候选元素时，再做最后确认

这也是它被称为“平衡版”的原因。

* 为什么循环条件是 `1 < j - i`
* 为什么平衡版中 `i = m` 不能写成 `i = m + 1`
* 为什么平衡版要把最终判断放到循环外
* 空数组时为什么要额外处理
* 如何手推一轮完整的平衡版二分查找

## 一眼先记住平衡版的本质

> 平衡版维护的是左闭右开区间 `[i, j)`；循环的目标不是“立刻找到答案”，而是不断缩小候选区间，直到只剩 1 个候选元素，再在循环外统一判断。

如果这句话记住了，后面的代码就不会觉得乱。

---

# 一、为什么叫“平衡版”？

“平衡版”这个名字，不是说它时间复杂度更低。
它的时间复杂度仍然是：

```text
O(log n)
```

所谓“平衡”，指的是：

> **它尽量让循环内每一轮的判断结构更均匀。**

---

## 先看基础版为什么“不平衡”

在理解平衡版之前，先回头看一下基础版的写法：

```java
if (target < a[m]) {
    ...
} else if (a[m] < target) {
    ...
} else {
    return m;
}
```

基础版的特点是：**每一轮不仅要缩小区间，还尝试在这一轮直接找到答案。**

这会带来一个现象：

* 如果目标落在左侧路径，那么很多轮次里只要第 1 个条件 `target < a[m]` 成立，就可以直接向左收缩
* 如果目标落在右侧路径，那么很多轮次里往往要先判断 `target < a[m]` 为假，再继续判断 `a[m] < target` 为真，才能向右收缩

也就是说，在**尚未命中目标的那些收缩轮次里**：

* 向左收缩，通常只需要 **1 次大小比较**
* 向右收缩，通常需要 **2 次大小比较**

因此，基础版所谓的“不平衡”，并不是说它左右两边的时间复杂度不同，而是说：

> **循环内部的比较成本不对称：左侧路径通常更省比较，右侧路径通常要做更多比较。**

如果把“不断缩小区间的轮数”记为 `L`，可以做一个直观理解：

* 目标偏左时，比较成本往往更接近 `L`
* 目标偏右时，比较成本往往更接近 `2L`

不过这里的 `L` 与 `2L` 只是为了帮助理解“左右路径比较开销不对称”这一点，**不应把它当成绝对精确的总比较次数公式**。因为在最终命中的那一轮，通常还会涉及额外的相等判断。

平衡版要做的事情，就是尽量削弱这种“左边便宜、右边较贵”的不对称，让循环内的判断结构更均匀。

---

## 平衡版怎么做？

平衡版把目标改成：

> **循环里只做一件事：继续缩小候选区间。**

因此它往往写成：

```java
if (target < a[m]) {
    j = m;
} else {
    i = m;
}
```

这里没有在循环内直接返回。
它把“是不是刚好相等”这件事延后，等区间缩到只剩一个候选元素时，再统一检查。

这样做的效果是：

* 循环内结构更简单
* 左右两边的处理更对称
* 平均比较过程更均衡

这就是“平衡版”名字的来源。

---

# 二、平衡版的区间模型是什么？

平衡版维护的仍然是 **左闭右开区间**：

```text
[i, j)
```

它表示：

* `i` 指向的元素属于候选范围
* `j` 只是右边界，不属于候选范围

所以候选元素实际覆盖的是：

```text
i ~ j - 1
```

这点和平衡版、改动版是一致的。

---

## 那平衡版和改动版差在哪？

虽然两者都用左闭右开区间 `[i, j)`，但它们的目标不同：

### 改动版

* 每一轮仍然试图直接找到答案
* 循环内发现相等就立刻 `return`

### 平衡版

* 循环内不急着返回
* 主要任务是把区间缩小到只剩一个候选元素
* 最后再统一比较这个候选元素是不是目标

所以两者的差别不在区间模型，而在 **循环策略**。

---

# 三、平衡版的核心思路

平衡版的思路可以浓缩成三句话：

1. 候选区间使用 `[i, j)`
2. 循环一直缩区间，直到只剩 1 个候选元素
3. 最后在循环外判断这个候选元素是不是目标

也就是说，平衡版把“找答案”拆成两段：

* 循环内：缩区间
* 循环外：做最后确认

---

# 四、为什么循环条件是 `1 < j - i`？

这是平衡版最关键的一行。

```java
while (1 < j - i)
```

这行如果没看懂，整篇都会觉得很绕。

---

## `j - i` 代表什么？

在左闭右开区间 `[i, j)` 中，区间里的元素个数正好是：

```text
j - i
```

例如：

* `[0, 8)` 里有 8 个元素
* `[3, 6)` 里有 3 个元素
* `[4, 5)` 里有 1 个元素
* `[4, 4)` 里有 0 个元素

所以：

> `j - i` 不是索引差而已，它正好就是“当前候选元素个数”。

---

## 那 `1 < j - i` 又代表什么？

它表示：

> 当前候选元素个数 **大于 1**。

也就是说，只要区间里还剩至少 2 个候选元素，就继续缩小。

一旦只剩下：

```text
j - i == 1
```

就说明区间里只剩最后一个候选元素，此时退出循环。

---

## 为什么不用 `i < j`？

因为：

* `i < j` 只说明区间非空
* 但平衡版想要的不是“只要非空就继续”
* 它想要的是“只要候选元素超过 1 个，就继续缩小”

所以平衡版的循环条件更精细：

* 改动版：区间非空就继续
* 平衡版：区间内候选元素 **超过 1 个** 才继续

这就是：

```java
while (1 < j - i)
```

的真正含义。

---

# 五、为什么平衡版要把最终判断放到循环外？

因为平衡版的设计目标就是：

> **先把区间缩到最小，再统一判断。**

当循环结束时，理想状态是：

```text
j - i == 1
```

也就是区间变成：

```text
[i, i + 1)
```

这表示区间里只剩一个候选元素：`a[i]`

此时只需要在循环外判断一次：

```java
target == a[i]
```

就知道答案了。

所以平衡版常见写法是：

```java
return target == a[i] ? i : -1;
```

但这里有一个前提：

> **数组不能为空。**

这一点非常重要，后面会专门讲。

---

# 六、为什么平衡版中是 `i = m`，而不是 `i = m + 1`？

这是平衡版最容易混淆的地方，也是最值得记住的地方。

先看代码：

```java
if (target < a[m]) {
    j = m;
} else {
    i = m;
}
```

很多人会问：

> 如果不是去左边，为什么不直接写 `i = m + 1`？

答案是：

> 因为在平衡版里，`a[m]` 仍然可能就是目标值，所以不能跳过 `m`。

---

## 关键原因：平衡版循环内不直接判等返回

在改动版里，你会写：

```java
if (target < a[m]) {
    j = m;
} else if (a[m] < target) {
    i = m + 1;
} else {
    return m;
}
```

这里如果走到“右边”，说明已经确认：

```text
a[m] < target
```

既然 `a[m]` 比目标小，那 `m` 就不可能是答案，所以可以放心写：

```java
i = m + 1;
```

---

## 但平衡版不同

平衡版里写的是：

```java
if (target < a[m]) {
    j = m;
} else {
    i = m;
}
```

注意这里的 `else` 并不表示：

```text
a[m] < target
```

它表示的是：

```text
target >= a[m]
```

也就是说：

* 目标可能在 `m` 的右边
* 目标也可能 **刚好就是 `m`**

既然 `m` 仍然可能是答案，就绝对不能跳过它。

所以只能写：

```java
i = m;
```

不能写：

```java
i = m + 1;
```

---

## 一句话记住这个区别

> 改动版里写 `i = m + 1`，是因为已经确认 `a[m] < target`；
> 平衡版里写 `i = m`，是因为只确认了 `target >= a[m]`，而 `m` 仍可能是目标。

这句话最好直接背下来。

---

# 七、平衡版代码模板

先给出一个 **可直接使用、且补了空数组处理** 的版本：

```java
/**
 * 二分查找平衡版
 *
 * @param a      待查找的升序数组
 * @param target 待查找的目标值
 * @return 找到则返回索引，找不到返回 -1
 */
public static int binarySearchBalance(int[] a, int target) {
    if (a.length == 0) {
        return -1;
    }

    int i = 0;
    int j = a.length;

    while (1 < j - i) {
        int m = (i + j) >>> 1;
        if (target < a[m]) {
            j = m;
        } else {
            i = m;
        }
    }

    return a[i] == target ? i : -1;
}
```

---

# 八、为什么这版必须补空数组判断？

这是原本很多平衡版讲解会漏掉的地方。

如果数组为空：

```java
int[] a = {};
```

那初始化后：

```java
i = 0;
j = 0;
```

此时：

```java
1 < j - i
```

也就是：

```text
1 < 0
```

条件不成立，循环不会执行。

如果你直接写：

```java
return a[i] == target ? i : -1;
```

那就会访问：

```java
a[0]
```

直接数组越界。

所以平衡版要么：

* 一开始单独处理空数组
* 要么改写整体逻辑，避免最后直接访问 `a[i]`

最常见也最直观的做法，就是：

```java
if (a.length == 0) {
    return -1;
}
```

# 九、手推一遍：查找 `38`

数组如下：

```java
int[] a = {7, 13, 21, 30, 38, 44, 52, 53};
int target = 38;
```

初始区间：

```text
[i, j) = [0, 8)
```

候选元素个数：

```text
j - i = 8
```

因为 `8 > 1`，继续循环。

---

## 第 1 轮

中间索引：

```text
m = floor((0 + 8) / 2) = 4
```

中间元素：

```text
a[4] = 38
```

判断：

```text
target < a[m]
即 38 < 38
```

结果为假，所以走 `else`：

```java
i = m;
```

变成：

```text
i = 4
```

新区间：

```text
[4, 8)
```

注意，这里没有直接返回。
因为平衡版的目标不是“循环内立刻结束”，而是继续缩区间。

---

## 第 2 轮

当前区间：

```text
[4, 8)
```

中间索引：

```text
m = floor((4 + 8) / 2) = 6
```

中间元素：

```text
a[6] = 52
```

判断：

```text
38 < 52
```

成立，所以：

```java
j = m;
```

新区间变成：

```text
[4, 6)
```

---

## 第 3 轮

当前区间：

```text
[4, 6)
```

中间索引：

```text
m = floor((4 + 6) / 2) = 5
```

中间元素：

```text
a[5] = 44
```

判断：

```text
38 < 44
```

成立，所以：

```java
j = m;
```

新区间变成：

```text
[4, 5)
```

---

## 结束

这时：

```text
j - i = 5 - 4 = 1
```

说明区间里只剩一个候选元素。

循环结束后，统一判断：

```java
a[i] == target
```

也就是：

```java
a[4] == 38
```

成立，返回 `4`。

---

## 这一轮手推最该记住什么？

最该记住的是：

> 即使第 1 轮已经命中 `a[4] == 38`，平衡版也不会立刻返回，而是继续把区间收缩到只剩一个候选元素，再统一确认。

这正是它和改动版最大的不同。

# 十一、平衡版与基础版、改动版的差异

你可以直接抓这三条主线。

---

## 1. 和基础版的差异

### 基础版

* 使用闭区间 `[i, j]`
* 循环条件是 `i <= j`
* 循环内可能直接返回
* 向右收缩写 `i = m + 1`

### 平衡版

* 使用左闭右开区间 `[i, j)`
* 循环条件是 `1 < j - i`
* 循环内主要负责缩区间
* 向右保留候选时写 `i = m`

---

## 2. 和改动版的差异

### 改动版

* 使用左闭右开区间 `[i, j)`
* 循环条件是 `i < j`
* 循环内会直接判等返回
* 因为右走前已确认 `a[m] < target`，所以写 `i = m + 1`

### 平衡版

* 也使用左闭右开区间 `[i, j)`
* 但循环条件是 `1 < j - i`
* 循环内不直接返回
* 因为只确认 `target >= a[m]`，`m` 仍可能是答案，所以写 `i = m`

---

# 十二、时间复杂度与空间复杂度

平衡版每一轮都会把候选区间缩小到原来的一半左右，因此时间复杂度仍然是：

```text
O(log n)
```

空间复杂度仍然是：

```text
O(1)
```

因为只使用了常数个变量：`i`、`j`、`m`。

# 十四、常见错误总结

## 错误 1：忘了处理空数组

问题：

* 循环可能直接跳过
* 最后访问 `a[i]` 时数组越界

---

## 错误 2：把 `i = m` 写成 `i = m + 1`

问题：

* 会把仍可能是答案的 `m` 跳掉
* 破坏平衡版“保留候选”的核心思想

---

## 错误 3：把循环条件写成 `i < j`

问题：

* 这会变成“区间非空就继续”的思路
* 失去平衡版“只剩 1 个候选时退出”的结构

---

## 错误 4：以为平衡版比基础版更快很多

问题：

* 它们的大 O 复杂度都还是 `O(log n)`
* 平衡版强调的是循环内判断结构更均衡，不是复杂度层级发生变化

---

## 错误 5：把平衡版和改动版混在一起

问题：

* 两者都用 `[i, j)`，很容易误以为只是小改
* 实际上它们最大的差别在于：**是否在循环内直接判等返回**

---

# 十五、一句话抓核心

> 平衡版二分查找维护的是左闭右开区间 `[i, j)`；循环内不急着直接返回，而是不断缩小区间，直到只剩 1 个候选元素；由于 `m` 仍可能是目标，所以右侧更新写 `i = m`，最后再在循环外统一判断。

# 十六、极简回查版

如果你以后只想用 20 秒回忆起平衡版，可以只看这段：

```java
public static int binarySearchBalance(int[] a, int target) {
    if (a.length == 0) {
        return -1;
    }

    int i = 0, j = a.length;
    while (1 < j - i) {
        int m = (i + j) >>> 1;
        if (target < a[m]) {
            j = m;
        } else {
            i = m;
        }
    }
    return a[i] == target ? i : -1;
}
```

配套记忆句：

* 区间是 `[i, j)`
* `j - i` 就是候选元素个数
* 只要候选数大于 1，就继续循环
* 这里只确认 `target < a[m]` 或 `target >= a[m]`
* 所以右走时不能跳过 `m`，必须写 `i = m`
