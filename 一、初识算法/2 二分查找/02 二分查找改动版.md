# 二分查找改动版

> - 所属章节：[一、初识算法 / 二分查找](./README.md)
> - 关键字：二分查找、左闭右开区间、`[i, j)`、`i < j`、`j = a.length`、边界收缩、死循环
> - 建议回查情境：想比较基础版与改动版的差异时、忘了为什么右边界是 `a.length` 时、分不清 `j = m` 和 `j = m - 1` 时、想弄懂为什么 `i <= j` 会死循环时

---

## 本节导读

这一节讲的是 **二分查找的改动版**。

它和基础版最大的不同，不是代码长得不一样，而是：

> **它维护的查找区间不再是闭区间 `[i, j]`，而是左闭右开区间 `[i, j)`。**

这件事一改变，后面很多写法都会一起变：

* `j` 的初值变成 `a.length`
* 循环条件从 `i <= j` 变成 `i < j`
* 向左收缩时从 `j = m - 1` 变成 `j = m`

所以这篇真正要学会的，不是“另一个模板”，而是：

> **区间模型一变，循环条件、边界含义、收缩方式都会跟着变。**

---

## 你会在这篇学到什么

* 什么是左闭右开区间 `[i, j)`
* 改动版和基础版的本质差异
* 为什么 `j` 初始化为 `a.length`
* 为什么循环条件必须写成 `i < j`
* 为什么改动版向左收缩时写 `j = m`
* 为什么误写成 `i <= j` 可能导致死循环
* 如何手推一轮完整的改动版二分查找

---

## 先回顾：基础版的区间模型是什么？

基础版维护的是：

```text
[i, j]
```

也就是闭区间。

含义是：

* `i` 指向的元素在候选范围内
* `j` 指向的元素也在候选范围内

所以基础版里：

* `i` 可能是目标
* `j` 也可能是目标

因此它使用：

```java
while (i <= j)
```

这是因为当 `i == j` 时，还剩最后一个候选元素要比较。

---

## 改动版最核心的变化：区间改成 `[i, j)`

改动版维护的是：

```text
[i, j)
```

这是一个 **左闭右开区间**，意思是：

* `i` 指向的元素属于候选范围
* `j` 只是右边界
* `a[j]` 不属于候选范围

这句话非常关键，可以直接决定整篇的所有写法。

你可以把 `[i, j)` 理解成：

> 候选元素的索引范围是 `i` 到 `j - 1`

也就是说：

* 左边界是“含进去的”
* 右边界是“卡住范围但不含进去的”

---

## 一眼先记住改动版的本质

> 改动版不是把基础版随便改几行，而是把查找区间从闭区间 `[i, j]` 换成左闭右开区间 `[i, j)`；因此 `j` 不再是候选元素，循环条件必须改成 `i < j`，向左收缩时也必须写成 `j = m`。

如果这句话记住了，整篇基本就不会乱。

---

# 一、为什么 `j = a.length`？

这是很多人第一次看改动版最不习惯的地方。

基础版写的是：

```java
int j = a.length - 1;
```

改动版却写成：

```java
int j = a.length;
```

看起来像“多加了一格”，其实不是乱写，而是因为区间模型已经变了。

---

## 因为这里的 `j` 不代表最后一个元素索引

在 `[i, j)` 这种写法里：

* `i` 是候选区间的起点
* `j` 是候选区间的“右边界外侧”

也就是说：

> `j` 指向的位置本来就不属于候选范围。

所以当整个数组都还没被排除时，候选范围应该是：

```text
[0, a.length)
```

这正好表示：

* 包含索引 `0`
* 不包含索引 `a.length`
* 实际上覆盖的是 `0 ~ a.length - 1`

这就刚好是整个数组。

所以改动版初始化一定写成：

```java
int i = 0;
int j = a.length;
```

---

## 你可以这样理解

假设数组长度为 `8`：

```java
int[] a = {7, 13, 21, 30, 38, 44, 52, 53};
```

索引范围是：

```text
0 1 2 3 4 5 6 7
```

那整个数组对应的左闭右开区间就是：

```text
[0, 8)
```

因为它包含 `0 ~ 7`，但不包含 `8`。

这就是为什么改动版的右边界初值不是 `7`，而是 `8`。

---

# 二、为什么循环条件是 `i < j`？

因为在左闭右开区间 `[i, j)` 中：

* 当 `i < j` 时，区间中至少还有一个候选元素
* 当 `i == j` 时，区间为空

这一点和基础版完全不同。

---

## 基础版中 `i == j` 的含义

基础版用的是闭区间 `[i, j]`，所以：

```text
[i, i]
```

里面还有一个元素。

因此基础版必须继续查。

---

## 改动版中 `i == j` 的含义

改动版用的是左闭右开区间 `[i, j)`，所以：

```text
[i, i)
```

这是一个 **空区间**。

因为它“从 `i` 开始，到 `i` 之前结束”，中间根本没有元素。

所以改动版里：

> `i == j` 不表示“只剩一个元素”，而表示“已经没有元素可查”。

因此循环条件必须写成：

```java
while (i < j)
```

---

## 这句话一定要记住

> 在闭区间 `[i, j]` 中，`i == j` 代表还剩 1 个元素。
> 在左闭右开区间 `[i, j)` 中，`i == j` 代表剩 0 个元素。

这就是基础版和改动版最容易混淆、也最值得记住的区别。

---

# 三、为什么改动版向左收缩时是 `j = m`？

这是这篇最关键的技术点之一。

改动版代码如下：

```java
public static int binarySearchAlternative(int[] a, int target) {
    int i = 0, j = a.length;
    while (i < j) {
        int m = (i + j) >>> 1;
        if (target < a[m]) {
            j = m;
        } else if (a[m] < target) {
            i = m + 1;
        } else {
            return m;
        }
    }
    return -1;
}
```

很多人看到这里会问：

> 为什么这里不是 `j = m - 1`？

答案是：

> 因为改动版的右边界 `j` 本来就不是候选元素。

---

## 情况 1：`target < a[m]`

说明目标只可能在左边。

但这里的“左边”在左闭右开模型里，不是：

```text
[i, m - 1]
```

而是：

```text
[i, m)
```

因为：

* `a[m]` 已经比较过，不是目标
* `m` 本身不该再纳入新区间
* 左闭右开的右边界本来就“不含进去”

因此直接写：

```java
j = m;
```

就已经准确表示新区间是：

```text
[i, m)
```

不需要再减一。

---

## 为什么 `j = m` 不会把 `m` 留下来？

这是很多人卡住的地方。

因为他们会不自觉用闭区间思维去看 `j = m`，以为“`m` 还在区间里”。

但在左闭右开区间里：

```text
[i, m)
```

右端点 `m` 是 **不包含** 的。

所以：

> `j = m` 并不是“把 `m` 留下来”，而是“把区间收缩到 `m` 之前”。

这正是左闭右开写法最容易绕进去、但一旦想通就很顺的地方。

---

# 四、为什么向右收缩时还是 `i = m + 1`？

因为当：

```java
a[m] < target
```

说明：

* `m` 不是目标
* `m` 左边也不可能是目标
* 目标只可能在 `m` 右边

所以新区间应当变成：

```text
[m + 1, j)
```

因此写成：

```java
i = m + 1;
```

这个和基础版是一致的。

---

## 所以改动版的左右收缩规则是

* 向左找：`j = m`
* 向右找：`i = m + 1`

你可以把它背成一句：

> 左闭右开里，右边界不含，所以左缩写 `j = m`；
> `m` 比较过后不能再留在左边界里，所以右缩还是 `i = m + 1`。

---

# 五、改动版代码模板

```java
/**
 * 二分查找改动版（左闭右开）
 *
 * @param a      待查找的升序数组
 * @param target 待查找的目标值
 * @return 找到则返回索引，找不到返回 -1
 */
public static int binarySearchAlternative(int[] a, int target) {
    int i = 0;
    int j = a.length;

    while (i < j) {
        int m = (i + j) >>> 1;

        if (target < a[m]) {
            j = m;
        } else if (a[m] < target) {
            i = m + 1;
        } else {
            return m;
        }
    }

    return -1;
}
```

# 六、手推一遍：查找 `38`

数组如下：

```java
int[] a = {7, 13, 21, 30, 38, 44, 52, 53};
int target = 38;
```

初始区间是整个数组：

```text
[i, j) = [0, 8)
```

---

## 第 1 轮

当前区间：

```text
[0, 8)
```

中间索引：

```text
m = floor((0 + 8) / 2) = 4
```

对应元素：

```text
a[4] = 38
```

比较结果：

```text
a[4] == target
```

直接返回：

```text
4
```

这个例子一步就找到，但它还不够说明边界怎么移动。

所以再看一个找不到的例子。

---

# 七、手推一遍：查找 `35`

数组：

```java
int[] a = {7, 13, 21, 30, 38, 44, 52, 53};
int target = 35;
```

初始区间：

```text
[i, j) = [0, 8)
```

---

## 第 1 轮

中间索引：

```text
m = 4
```

中间元素：

```text
a[4] = 38
```

因为：

```text
35 < 38
```

所以目标只可能在左边。

新区间变成：

```text
[0, 4)
```

也就是：

```java
j = m;
```

---

## 第 2 轮

当前区间：

```text
[0, 4)
```

中间索引：

```text
m = 2
```

中间元素：

```text
a[2] = 21
```

因为：

```text
21 < 35
```

所以目标只可能在右边。

新区间变成：

```text
[3, 4)
```

也就是：

```java
i = m + 1;
```

---

## 第 3 轮

当前区间：

```text
[3, 4)
```

中间索引：

```text
m = 3
```

中间元素：

```text
a[3] = 30
```

因为：

```text
30 < 35
```

所以目标只可能在右边。

新区间变成：

```text
[4, 4)
```

也就是：

```java
i = 4;
```

---

## 结束

这时：

```text
i == j == 4
```

区间变成：

```text
[4, 4)
```

这是空区间。

说明已经没有任何候选元素了，所以查找失败，返回 `-1`。

---

## 这一轮手推最该记住什么？

最该记住的是：

> 在改动版里，`[4, 4)` 不是“剩一个元素”，而是“一个元素都没有”。

所以改动版使用：

```java
while (i < j)
```

是完全合理的。

# 八、为什么不能写成 `while (i <= j)`？

这是改动版最容易出 bug 的地方。

很多人把基础版写熟后，会下意识写成：

```java
while (i <= j)
```

但在改动版里，这样写可能会死循环。

---

## 错误示例

```java
public static int binarySearchAlternativeWrong(int[] a, int target) {
    int i = 0;
    int j = a.length;

    while (i <= j) {
        int m = (i + j) >>> 1;

        if (target < a[m]) {
            j = m;
        } else if (a[m] < target) {
            i = m + 1;
        } else {
            return m;
        }
    }

    return -1;
}
```

---

## 为什么会死循环？

关键场景是：

```text
i == j
```

在左闭右开区间中，这本来已经是空区间了，根本不应该再进循环。

但如果你写成 `i <= j`，程序还是会继续执行。

更糟的是，有些情况下会出现：

* `i == j == m`
* 且 `target < a[m]`

这时执行：

```java
j = m;
```

结果就是：

```text
j 没变
```

同时 `i` 也没变。

也就是说，新一轮还是同样的：

```text
i、j、m
```

区间完全没有缩小。

程序会不断重复这一轮，最后卡死。

---

## 用一句话说清死循环机制

> 改动版的收缩方式依赖“区间非空时才继续循环”；
> 一旦把条件误写成 `i <= j`，就会在空区间或边界重合时继续执行，而 `j = m` 又可能让区间无法继续缩小，于是产生死循环。

---

# 九、改动版和基础版到底差在哪里？

你可以直接看这张对照：

## 1. 区间模型不同

* 基础版：`[i, j]`
* 改动版：`[i, j)`

## 2. `j` 的含义不同

* 基础版：`j` 可能指向目标
* 改动版：`j` 只是右边界，不指向目标

## 3. 循环条件不同

* 基础版：`i <= j`
* 改动版：`i < j`

## 4. 向左收缩方式不同

* 基础版：`j = m - 1`
* 改动版：`j = m`

## 5. 向右收缩方式

* 两者都写：`i = m + 1`

---

# 十、改动版的时间复杂度

每一轮都会排除掉大约一半候选区间，因此时间复杂度仍然是：

```text
O(log n)
```

空间复杂度仍然是：

```text
O(1)
```

因为只使用了常数个变量：`i`、`j`、`m`。

# 十二、常见错误总结

## 错误 1：把 `j` 初始化成 `a.length - 1`

问题：

* 这会把左闭右开区间写成四不像
* 代码表面像改动版，实际区间语义已经乱掉

---

## 错误 2：把循环条件写成 `i <= j`

问题：

* `i == j` 在改动版中已经是空区间
* 不该继续循环
* 某些情况下会死循环

---

## 错误 3：把 `j = m` 写成 `j = m - 1`

问题：

* 这是把闭区间的写法硬套到左闭右开
* 会破坏 `[i, j)` 的区间语义

---

## 错误 4：没搞清 `j` 是否是候选元素

问题：

* 一旦这件事没想清楚
* 整个改动版的循环条件和边界更新都会跟着乱

---

# 十三、一句话抓核心

> 改动版二分查找维护的是左闭右开区间 `[i, j)`；`i` 指向的元素仍是候选，`j` 只是右边界而不是候选元素；因此循环条件是 `i < j`，向左收缩写 `j = m`，向右收缩写 `i = m + 1`。
