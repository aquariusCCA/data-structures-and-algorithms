## 本节导读

这一节讲的是 **二分查找的基础版**。

很多人第一次学二分查找，容易把注意力放在“代码长什么样”，结果背了模板，却不明白为什么这样写。
其实基础版真正要掌握的，只有下面四件事：

1. 查找区间是 **闭区间 `[i, j]`**
2. `i`、`j`、`m` 都可能指向目标元素
3. 只要区间里还有元素，就必须继续比较，所以循环条件是 `i <= j`
4. 每次比较完 `m` 后，都要把 `m` 排除掉，因此边界更新是 `m - 1` 或 `m + 1`

如果这四件事真的想通，二分查找后面的改动版、平衡版、Leftmost / Rightmost 都会顺很多。

## 什么是二分查找？

二分查找（Binary Search），也叫折半查找，是一种在 **有序数组** 中查找目标值的高效算法。

它的核心思想很简单：

* 不从左到右一个一个找
* 而是每次直接看中间元素
* 再根据大小关系，马上排除掉一半不可能的区域

也就是说，它依赖的前提不是“数组存在”，而是：

> **数组必须有序。**

如果数组无序，就不能根据中间值判断目标到底在左边还是右边，二分查找也就失效了。

---

## 基本需求

需求：在一个 **升序数组** `a` 中查找 `target`

* 如果找到，返回目标值所在索引
* 如果找不到，返回 `-1`

例如：

```java
int[] a = {7, 13, 21, 30, 38, 44, 52, 53};
```

* 查找 `30`，返回 `3`
* 查找 `38`，返回 `4`
* 查找 `15`，返回 `-1`

---

## 先建立一个核心模型：闭区间 `[i, j]`

基础版二分查找维护的是一个 **闭区间**：

```text
[i, j]
```

它表示：

* `i` 是左边界，且 `a[i]` 仍然属于候选范围
* `j` 是右边界，且 `a[j]` 也仍然属于候选范围

这句话非常关键，因为它直接决定了后面三个东西：

1. 为什么 `i == j` 时还要继续比较
2. 为什么循环条件是 `i <= j`
3. 为什么缩区间时要写 `j = m - 1`、`i = m + 1`

你可以把它理解成：

> `[i, j]` 里面的每个元素，都是“还没有被排除的候选人”。

---

## 严谨一点的算法描述

前提：给定一个长度为 `n` 的升序数组 `a`，满足：

```text
a[0] <= a[1] <= a[2] <= ... <= a[n - 1]
```

以及待查值 `target`

算法过程如下：

1. 初始化左边界 `i = 0`，右边界 `j = n - 1`
2. 当 `i <= j` 时，说明闭区间 `[i, j]` 中仍有候选元素，继续查找
3. 计算中间索引

```text
m = floor((i + j) / 2)
```

4. 若 `target < a[m]`，说明目标只可能在左半边，令 `j = m - 1`
5. 若 `a[m] < target`，说明目标只可能在右半边，令 `i = m + 1`
6. 若 `a[m] == target`，返回 `m`
7. 若循环结束仍未找到，返回 `-1`

---

## 基础版代码模板

```java
/**
 * 二分查找基础版
 *
 * @param a      待查找的升序数组
 * @param target 待查找的目标值
 * @return 找到则返回索引，找不到返回 -1
 */
public static int binarySearchBasic(int[] a, int target) {
    int i = 0;
    int j = a.length - 1;

    while (i <= j) {
        int m = (i + j) >>> 1;

        if (target < a[m]) {
            j = m - 1;
        } else if (a[m] < target) {
            i = m + 1;
        } else {
            return m;
        }
    }

    return -1;
}
```

---

## 一眼先记住这句话

> 基础版二分查找使用闭区间 `[i, j]`；只要 `i <= j`，区间内就还有候选元素；每次比较中间位置 `m` 后，都排除掉 `m` 以及一整半不可能区域。

如果你以后忘了细节，只要先想起这句话，代码通常就能自己推回来。

# 一、为什么基础版使用闭区间 `[i, j]`？

因为基础版把左右边界都看成 **有效候选位置**。

也就是说：

* `i` 可能是答案
* `j` 可能是答案
* 中间位置 `m` 当然也可能是答案

所以基础版的本质是：

> **边界本身也参与查找。**

这就是它和左闭右开写法最大的不同。

---

# 二、为什么循环条件是 `i <= j`？

这是基础版最重要的一点。

## 1. `i < j` 表示什么？

表示区间里至少还有两个元素。

## 2. `i == j` 表示什么？

表示区间里还剩下 **最后一个元素**。

例如：

```text
[i, j] = [4, 4]
```

这不是空区间，而是还有一个候选元素 `a[4]`。

而且这一轮算出来的中间索引一定是：

```java
m = i = j
```

所以这个元素仍然必须比较。

## 3. `i > j` 才表示什么？

才表示区间为空，也就是：

> 已经没有任何元素可能是目标值了。

所以基础版的正确循环条件一定是：

```java
while (i <= j)
```

而不是：

```java
while (i < j)
```

---

## 为什么 `i < j` 会漏掉答案？

看一个最简单的例子：

```java
int[] a = {1, 3};
int target = 3;
```

如果写成：

```java
while (i < j)
```

那当区间缩小到只剩一个元素时：

```java
i == j
```

循环会直接结束。

问题在于：

> 这个最后剩下的元素，还没有比较。

也就是说，程序在“最后一个候选人”还没接受检查之前，就提前宣布结束了。

这就可能漏掉答案。

---

## 错误示例

```java
public static int binarySearchWrong(int[] a, int target) {
    int i = 0, j = a.length - 1;

    while (i < j) {
        int m = (i + j) >>> 1;
        if (target < a[m]) {
            j = m - 1;
        } else if (a[m] < target) {
            i = m + 1;
        } else {
            return m;
        }
    }

    return -1;
}
```

这个版本的根本问题不是代码形式，而是：

> 它把“只剩一个元素”误判成“没有元素可查”。

---

# 三、为什么边界更新一定是 `m - 1` 和 `m + 1`？

因为 `m` 已经比较过了。

这一点也非常关键。

---

## 情况 1：`target < a[m]`

说明：

* `m` 位置不是目标
* `m` 右边更不可能是目标
* 目标只可能出现在左半边

所以新区间应该变成：

```text
[i, m - 1]
```

因此写成：

```java
j = m - 1;
```

---

## 情况 2：`a[m] < target`

说明：

* `m` 位置不是目标
* `m` 左边更不可能是目标
* 目标只可能出现在右半边

所以新区间应该变成：

```text
[m + 1, j]
```

因此写成：

```java
i = m + 1;
```

---

## 为什么不能写成 `j = m` 或 `i = m`？

因为那样会把已经比较过的 `m` 留在候选区间里。

这会导致两个问题：

1. 重复比较同一个位置
2. 某些情况下无法缩小区间，甚至死循环

所以基础版一定要记住：

> **比较过 `m` 之后，必须把 `m` 排除。**

---

# 四、手推一遍：查找 `38`

数组如下：

```java
int[] a = {7, 13, 21, 30, 38, 44, 52, 53};
int target = 38;
```

---

## 第 1 轮

初始区间：

```text
[i, j] = [0, 7]
```

中间索引：

```text
m = floor((0 + 7) / 2) = 3
```

对应元素：

```text
a[3] = 30
```

比较：

```text
38 > 30
```

说明目标在右边，更新左边界：

```text
i = m + 1 = 4
```

新区间变成：

```text
[4, 7]
```

---

## 第 2 轮

当前区间：

```text
[i, j] = [4, 7]
```

中间索引：

```text
m = floor((4 + 7) / 2) = 5
```

对应元素：

```text
a[5] = 44
```

比较：

```text
38 < 44
```

说明目标在左边，更新右边界：

```text
j = m - 1 = 4
```

新区间变成：

```text
[4, 4]
```

---

## 第 3 轮

当前区间：

```text
[i, j] = [4, 4]
```

注意，这里只剩下一个元素，但它仍然是候选元素。

中间索引：

```text
m = 4
```

对应元素：

```text
a[4] = 38
```

比较结果：

```text
a[4] == target
```

返回：

```text
4
```

---

## 这一轮手推最该记住什么？

最该记住的是：

> 当区间变成 `[4, 4]` 时，仍然要继续比较。
> 这就是为什么基础版必须写 `i <= j`。

如果这里写成 `i < j`，第三轮根本不会执行，答案就漏掉了。

---

# 五、为什么 `(i + j) / 2` 可能有问题？

很多人一开始会写：

```java
int m = (i + j) / 2;
```

从逻辑上看没有错，但从实现角度看，它有一个潜在风险：

> `i + j` 可能发生整数溢出。

---

## 为什么会溢出？

假设：

```java
int i = 0;
int j = Integer.MAX_VALUE - 1;
```

第一轮可能没事。

但当 `i` 和 `j` 都变得很大时，再做：

```java
i + j
```

这个和有可能超过 `int` 的最大值 `2147483647`。

一旦超过，就会发生溢出，结果变成负数，最终导致中间索引错误。

---

# 六、更安全的中间索引写法

## 写法一：最稳、最容易解释

```java
int m = i + ((j - i) >>> 1);
```

它的意思是：

* 先算出区间长度 `j - i`
* 再取一半
* 最后从 `i` 向右移动这一半距离

优点很明显：

* 不直接计算 `i + j`
* 更不容易让人误解
* 工程上也很常见

---

## 写法二：Java 中也很常见

```java
int m = (i + j) >>> 1;
```

这也是标准库和很多源码里常见的写法。

但这里要注意一个表达精度问题：

### 不够严谨的说法

“`>>> 1` 可以避免溢出”

### 更严谨的说法

`i + j` 这一步本身仍可能在 `int` 范围内发生位级别溢出；
但在二分查找中索引非负的前提下，使用 `(i + j) >>> 1` 仍能正确得到中间位置。

所以更准确地说：

> 它不是让加法“完全不溢出”，而是即使发生了二进制层面的溢出，最终右移后的结果仍然能作为正确中点。

---

## 示例代码

```java
@Test
@DisplayName("测试中间索引计算")
public void testMid() {
    int i = 0;
    int j = Integer.MAX_VALUE - 1;

    int m1 = (i + j) >>> 1;
    System.out.println(m1);

    i = m1 + 1;
    int m2 = (i + j) >>> 1;
    System.out.println(m2);
}
```

# 七、为什么比较时常写成两个“小于号”？

常见写法如下：

```java
if (target < a[m]) {
    ...
} else if (a[m] < target) {
    ...
} else {
    ...
}
```

这种写法有两个好处。

## 1. 逻辑顺序自然

因为数组是升序的：

* `target < a[m]`：向左找
* `a[m] < target`：向右找
* 否则：相等

方向和大小关系正好一一对应，读起来很顺。

## 2. “相等”自然落到最后

前两种情况已经覆盖：

* 小于
* 大于

剩下的就只能是相等，所以用 `else` 很自然。

这比把三种情况都单独写出来更简洁。

---

# 八、时间复杂度为什么是 `O(log n)`？

每一轮比较后，都会排除掉大约一半元素。

例如：

* 原来有 `n` 个候选元素
* 下一轮大约剩 `n / 2`
* 再下一轮大约剩 `n / 4`
* 再下一轮大约剩 `n / 8`

一直减到只剩 1 个元素为止。

所以循环次数大约满足：

```text
n / 2^k = 1
```

解出来就是：

```text
k = log₂n
```

因此时间复杂度是：

```text
O(log n)
```

空间复杂度是：

```text
O(1)
```

因为只用了几个变量：`i`、`j`、`m`。

# 十、常见错误总结

## 错误 1：把循环条件写成 `i < j`

问题：

* 最后一个候选元素可能不会被比较
* 会漏答案

---

## 错误 2：把边界更新写成 `j = m` 或 `i = m`

问题：

* `m` 已经比较过了，却没有被排除
* 可能重复比较甚至死循环

---

## 错误 3：默认数组有序，却没写清前提

问题：

* 读者可能误以为任何数组都能二分查找
* 实际上二分查找只适用于有序数据

---

## 错误 4：把 `(i + j) >>> 1` 解释成“完全不会溢出”

问题：

* 说法不够严谨
* 更准确的理解是：即使加法发生了位级别溢出，在索引非负前提下仍能得到正确中点

---

# 十一、一句话抓核心

> 基础版二分查找维护的是闭区间 `[i, j]`；只要 `i <= j`，区间中就还有候选元素；每次比较中间元素后，排除掉 `m` 和一整半不可能区域，直到找到目标或区间为空为止。
