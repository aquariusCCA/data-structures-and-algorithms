# 一、什麼是遞歸

遞歸，就是**函式呼叫自己**來解決問題。

它通常適合這種情況：

* 大問題可以拆成**同類型的小問題**
* 小問題的解法和大問題的解法長得很像
* 可以找到一個**最小情況**，讓遞歸停止

---

# 二、為什麼遞歸難

遞歸難的原因不是語法，而是思維方式不直覺。

人比較習慣：

* 一步一步做
* 用變數追蹤過程
* 明確看到每一步怎麼走

但遞歸要求你這樣想：

> 我只處理當前這一層
> 剩下的交給下一層的自己處理

所以遞歸的關鍵不是硬模擬全部流程，而是先相信函式的定義。

---

# 三、寫遞歸時一定要先想的 3 件事

每一題都先問自己：

## 1. 這個函式代表什麼

先定義函式的語意。

例如：

* `sum(n)`：回傳 1 到 n 的總和
* `reverse(s)`：回傳字串 s 的反轉結果
* `gcd(a, b)`：回傳 a 和 b 的最大公因數

---

## 2. 什麼時候不用再拆

也就是 base case（終止條件）。

如果沒有終止條件，遞歸就會一直呼叫下去，最後 stack overflow。

---

## 3. 當前這一層要做什麼

當前這一層通常只做兩件事：

* 處理眼前的一小部分
* 把剩下的問題交給下一層

---

# 四、遞歸的核心模板

可以先用這個思考模板：

```java
// 1. 函式代表什麼？
// 2. 什麼情況直接回傳？
// 3. 當前這一層怎麼把問題縮小？
```

更具體一點就是：

```java
public static ReturnType func(參數) {
    // base case
    if (最小情況) {
        return 直接答案;
    }

    // recursive case
    return 當前處理 + func(更小的問題);
}
```

---

# 五、學遞歸最重要的觀念

## 1. 不要一開始就硬模擬全部過程

遞歸不是先把每一層都想完才寫得出來。

真正的做法是：

* 先定義函式的意思
* 相信下一層會把子問題處理好
* 你只要決定當前這一層怎麼接上去

---

## 2. 遞歸的本質是「縮小問題」

常見縮小方式有：

* `n -> n - 1`
* `index -> index + 1`
* `left/right -> left + 1, right - 1`
* `a, b -> b, a % b`

---

## 3. base case 要和函式語意一致

這點很重要。

例如：

* 如果 `max(arr, n)` 代表「前 n 個元素的最大值」
* 那 base case 應該是 `n == 1`
* 而不是 `n == 0` 回傳 `arr[0]`

也就是說：

> base case 不是隨便設一個能停下來的條件
> 而是要符合函式本身的定義

---

# 六、判斷一題能不能用遞歸的方式

看到題目時，可以先問：

## 1. 能不能拆成更小的同類問題？

如果可以，就很有遞歸味道。

## 2. 有沒有最小情況？

如果沒有停下來的條件，不能直接寫遞歸。

## 3. 當前這一層能不能只做局部決策？

如果可以，就適合遞歸。

---

現在做題時，應該開始習慣這樣思考：

## Step 1：先寫函式語意

例如：

```java
// digitSum(n)
// 表示回傳整數 n 各位數字的總和
```

---

## Step 2：寫終止條件

例如：

* 只剩一位數
* index 走到結尾
* left >= right
* b == 0
* n == 0 或 n == 1

---

## Step 3：寫遞歸式

把問題縮小。

# 九、之後做遞歸題時的固定流程

之後每次做題，先照這個順序：

## 1. 先寫一句函式語意

例如：

```java
// gcd(a, b) 表示 a 和 b 的最大公因數
```

## 2. 寫出 base case

例如：

```java
if (b == 0) {
    return a;
}
```

## 3. 想怎麼縮小問題

例如：

```java
return gcd(b, a % b);
```

# 十一、一句總結

遞歸的本質可以濃縮成一句話：

> 先定義函式在做什麼，處理好最小情況，然後把剩下的同類問題交給下一層
